# 4.6 - Static Variables

## Big idea

`static` has TWO completely different meanings in C, depending on WHERE
you use it. This lesson covers both.

```text
static INSIDE a function = remembers its value between function calls
static OUTSIDE a function = private to this .c file only (other files can't see it)
```

---

# PART 1 - static inside a function (usable RIGHT NOW, no prerequisites)

## The idea

A normal local variable resets every time a function is called. A
`static` local variable does NOT reset - it keeps its value between calls.

```c
void countCalls(void) {
    static int count = 0;   // only initialized ONCE, ever - the first call

    count++;

    printf("%d\n", count);
}
```

Calling `countCalls()` three times in a row:

```text
1
2
3
```

Compare this to a NORMAL local variable doing the same thing:

```c
void countCallsNormal(void) {
    int count = 0;   // resets to 0 EVERY single call

    count++;

    printf("%d\n", count);
}
```

Calling THIS three times in a row:

```text
1
1
1
```

## Why this matters

`static` local variables let a function "remember" something across
multiple calls, without needing an external variable (Ch.4.3) shared
with other functions. It's PRIVATE memory just for that one function,
that persists between calls.

```text
normal local variable  = fresh every call, forgotten when function ends
static local variable  = created ONCE, remembers its value forever after
external variable      = shared by EVERY function in the file
```

## What I need to remember (Part 1)

```text
static inside a function = keeps its value between calls

a static variable is only initialized ONCE - the very first time
the function runs - never again, even if the function is called
many more times

useful for counters, "has this run before" flags, or remembering
state across calls without using a shared external variable
```

## The rule

```text
static outside a function = this variable is PRIVATE to this .c file.
No other .c file can see or touch it, even with extern.
```

This is the SAME idea as an external variable (Ch.4.3) - shared across
functions in one file - but with an extra restriction: NO OTHER FILE
is allowed to reach it either.

## K&R's example - the calculator program

K&R imagines a calculator program split across several files:

```text
calc.h
main.c
getop.c
stack.c
getch.c
```

### stack.c - private stack data

```c
static int sp = 0;
static double val[MAXVAL];

void push(double f) {
    if (sp < MAXVAL) {
        val[sp] = f;
        sp++;
    } else {
        printf("Error: stack full\n");
    }
}

double pop(void) {
    if (sp > 0) {
        sp--;
        return val[sp];
    } else {
        printf("Error: stack empty\n");
        return 0.0;
    }
}
```

`sp` and `val` are marked `static` - ONLY functions inside `stack.c`
(like `push` and `pop`) can touch them directly. Other files (like
`main.c`) are FORBIDDEN from reaching `sp` or `val` directly - they
must go through `push()` and `pop()` instead.

```text
stack.c
│
├── private static variables
│   ├── sp
│   └── val
│
└── public functions
    ├── push
    └── pop
```

### getch.c - private input buffer

```c
static char buf[BUFSIZE];
static int bufp = 0;

// getch() and ungetch() use buf and bufp internally,
// but other files can only call getch()/ungetch() - never touch
// buf or bufp directly.
```

```text
getch.c
│
├── private static variables
│   ├── buf
│   └── bufp
│
└── public functions
    ├── getch
    └── ungetch
```

### Why this separation matters

Without `static`, ANY file in the whole program could reach in and
directly modify `sp`, `val`, `buf`, or `bufp` - which is dangerous,
since those files might not understand the internal rules those
variables depend on (like "sp must never go negative").

```text
main.c should NOT directly change sp
getop.c should NOT directly change bufp

Instead:
main.c uses push() and pop()
getop.c uses getch() and ungetch()
```

This is the same idea as Ch.4.3's warning about external variables
("use functions, not direct access, when possible") - just enforced
more strictly, at the FILE level instead of the function level.

## What I need to remember (Part 2 - for later)

```text
static outside a function = private to THIS .c file - other files
cannot see or touch it, even with extern

this is how K&R's calculator hides sp/val inside stack.c, and
buf/bufp inside getch.c

other files interact through PUBLIC functions (push, pop, getch,
ungetch) instead of touching private data directly

this concept fully depends on having multiple .c files AND arrays -
revisit after Ch.5 (arrays) and after learning header files (Ch.4.5)
```
