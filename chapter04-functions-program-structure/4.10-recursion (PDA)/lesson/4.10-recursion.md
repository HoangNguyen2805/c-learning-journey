# 4.10 - Recursion

## Big idea

Recursion means a function calls itself.

A recursive function always needs two parts:

```text
base case      = when to stop
recursive case = when the function calls itself again, on a smaller problem
```

If the base case is missing (or unreachable), the function calls itself
forever — which crashes the program. More on *why* it crashes below.

---

## Simple recursion example

```c
void countdown(int n) {
    if (n <= 0) {
        printf("Done\n");
    }
    else {
        printf("%d\n", n);
        countdown(n - 1);
    }
}
```

Calling `countdown(3);` prints:

```text
3
2
1
Done
```

---

## Base case

The base case stops the recursion:

```c
if (n <= 0) {
    printf("Done\n");
}
```

Without a base case, the function keeps calling itself with no way out.

---

## Recursive case

The recursive case makes progress **toward** the base case — each call must
move the problem closer to stopping, or it never will:

```c
countdown(n - 1);
```

```text
countdown(3)
countdown(2)
countdown(1)
countdown(0)   <- base case reached, stop
```

---

## Why recursion can crash: the call stack

Every function call — recursive or not — is placed on the **call stack**
when it starts, and removed when it finishes (returns). While a call is
still running (waiting on a call it made), it stays on the stack, holding
onto its own local variables and parameters.

For recursion, each call adds a new stack frame **before** the previous one
finishes. If the base case is never reached, calls keep piling onto the
stack indefinitely. The stack has a fixed size — eventually there's no room
left, and the program crashes. This is called a **stack overflow**.

```text
countdown(3) called      -> stack: [3]
  countdown(2) called    -> stack: [3, 2]
    countdown(1) called  -> stack: [3, 2, 1]
      countdown(0) called-> stack: [3, 2, 1, 0]   <- base case, starts returning
    countdown(1) returns -> stack: [3, 2, 1]
  countdown(2) returns   -> stack: [3, 2]
countdown(3) returns     -> stack: [3]
(main continues)         -> stack: []
```

A missing base case means this stack only ever grows, never shrinks —
that's the mechanism behind the crash, not just "it runs forever."

## Call stack trace example (worked in full)

This is the single most useful skill for understanding recursion: being
able to trace a call by hand, both going *down* (calls being made) and
coming back *up* (values resolving).

Using `factorial`:

```c
int factorial(int n) {
    if (n == 0)
        return 1;
    return n * factorial(n - 1);
}
```

Tracing `factorial(3)`:

**Going down (each call waits on the next):**

```text
factorial(3)
  -> needs factorial(2) to continue
factorial(2)
  -> needs factorial(1) to continue
factorial(1)
  -> needs factorial(0) to continue
factorial(0)
  -> base case: returns 1 immediately
```

**Coming back up (each call finishes using the value it was waiting on):**

```text
factorial(0) = 1
factorial(1) = 1 * factorial(0) = 1 * 1        = 1
factorial(2) = 2 * factorial(1) = 2 * 1        = 2
factorial(3) = 3 * factorial(2) = 3 * 2        = 6
```

So `factorial(3)` returns `6`, but the important part isn't the final
number — it's seeing that **nothing multiplies until the base case
returns first**, and then the answer builds back up one level at a time.

## K&R printd example

K&R uses recursion to print an integer one digit at a time, in the
correct left-to-right order — even though the *easiest* digit to compute
first (`n % 10`) is the *last* digit.

```c
void printd(int n) {
    if (n < 0) {
        putchar('-');
        n = -n;
    }
    if (n / 10) {
        printd(n / 10);
    }
    putchar(n % 10 + '0');
}
```

**Why the digits come out in the right order:**

The key line is:

```c
if (n / 10) {
    printd(n / 10);
}
putchar(n % 10 + '0');
```

The recursive call `printd(n / 10)` happens **before** the
`putchar(n % 10 + '0')` on that same level. That means every call goes as
deep as possible (peeling off digits) *before any digit is printed at
all*. Printing only starts happening as the calls return back up — so the
first `putchar` to actually run is the one for the leftmost digit.

Trace for `printd(123)`:

**Going down:**

```text
printd(123)  -> n/10 = 12, not 0, so recurse first
  printd(12) -> n/10 = 1, not 0, so recurse first
    printd(1)  -> n/10 = 0, stop recursing
      putchar('1')      <- prints '1' first
  putchar('2')           <- back in printd(12), prints '2'
putchar('3')              <- back in printd(123), prints '3'
```

**Result:** `1`, then `2`, then `3` are printed in that order → output is
`123`.

This is the exact same "unwind and act on the way back up" pattern as the
factorial trace above — the only difference is factorial *computes* a
value on the way back up, and `printd` *prints* something on the way back
up.

## Factorial example (full program)

```c
int factorial(int n) {
    if (n == 0) {
        return 1;
    }
    else {
        return n * factorial(n - 1);
    }
}
```

```text
5! = 5 * 4 * 3 * 2 * 1 = 120
```

Recursive definition:

```text
factorial(n) = n * factorial(n - 1)
factorial(0) = 1              (base case)
```

**Watch out:** this function has no protection against negative input.
`factorial(-1)` would call `factorial(-2)`, then `factorial(-3)`, and so
on — `n` never reaches `0`, so the base case is never hit. This is a
stack overflow waiting to happen. A more robust version would check for
`n < 0` and handle it explicitly (e.g. return an error code, or treat it
as invalid input) rather than assuming `n` is always non-negative.

## Recursion vs. iteration

Recursion and loops can often solve the same problem. Neither is
"better" universally:

```text
Recursion:
  + often clearer / more natural for problems that are already defined
    in terms of smaller versions of themselves (trees, divide-and-conquer,
    nested structures)
  - has function-call overhead (each call = a new stack frame)
  - risks stack overflow if the base case is wrong or missing

Iteration (loops):
  + no extra call overhead, no stack growth
  + often more efficient for simple repeated tasks (like countdown)
  - can be harder to read for problems that are naturally recursive
    (e.g. tree traversal)
```

`countdown` and `factorial` could both be written with a simple loop
instead. They're written recursively here because K&R uses them to
*teach* recursion — not because recursion is the only or best way to
solve them.

---

## Recursion is useful when...

A problem naturally breaks into smaller versions of itself:

```text
printing digits        (printd)
factorial
tree problems
sorting problems like quicksort
```

K&R also mentions quicksort as a recursive example — divide the array,
recursively sort each half, combine.

## What I need to remember

```text
recursion = function calls itself

base case      = stops recursion
recursive case = calls the function again with a smaller/simpler problem

without a base case, recursion keeps growing the call stack until it
overflows and crashes

each recursive call gets its own copy of local variables and parameters,
and stays on the call stack until it returns

work "on the way down" happens before the base case
work "on the way back up" happens after the base case, as each call
finishes and returns to the one that called it

printd's ordering trick: recurse BEFORE printing, so printing only
starts once the deepest call is reached and calls start returning

recursion vs. iteration is a tradeoff, not a rule — recursion shines when
the problem is naturally self-similar (trees, divide-and-conquer)

K&R uses printd and quicksort as recursion examples
```

---

## Simple summary

```text
4.10 recursion = solve a problem by calling the same function on a
smaller version of the problem, trusting that the smaller call will work
correctly (this trust is sometimes called the "leap of faith"), and
building up the final answer as calls return.
```
