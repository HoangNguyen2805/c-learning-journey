# Recursion as a Pushdown Automaton (PDA)

## Why this lesson exists

Today we traced `countdown`, `mystery`, and `factorial` by hand, one call
at a time. The core insight that came out of it: **a function call stack
behaves exactly like a PDA's stack** — strict push/pop, last-in-first-out,
no way to resolve something in the middle without resolving everything
above it first.

This lesson formalizes that connection.

---

## Quick PDA refresher

A pushdown automaton processes input and manages a **stack**, where:

```text
push  = place a new symbol on top of the stack
pop   = remove the top symbol from the stack
LIFO  = Last In, First Out — you can only ever act on the TOP symbol
```

You cannot reach into the middle of the stack. You cannot pop symbol #2
before symbol #3 (the one above it) has been popped. Order is strict.

---

## The call stack is a PDA's stack

Every time a function is called, it's **pushed** onto the call stack.
Every time a function finishes (returns), it's **popped** off.

```text
function call    -> push
function returns -> pop
```

Just like a PDA, you cannot "finish" `factorial(2)` before `factorial(1)`
(the thing above it, which it called) has finished and popped off first.

---

## Case 1: `countdown` — a stack with nothing left to do after popping

```c
void countdown(int n) {
    if (n <= 0) {
        printf("Done\n");
    }
    else {
        printf("%d\n", n);
        countdown(n - 1);
    }
}
```

Calling `countdown(2)`:

```text
PUSH countdown(2)   stack: [2]
PUSH countdown(1)   stack: [2, 1]
PUSH countdown(0)   stack: [2, 1, 0]   <- base case: prints "Done"
POP  countdown(0)   stack: [2, 1]      <- nothing left to do, just pops
POP  countdown(1)   stack: [2]         <- nothing left to do, just pops
POP  countdown(2)   stack: []          <- nothing left to do, just pops
```

Printed output, in order it actually happened: `2`, `1`, `Done`.

This is the **simplest PDA behavior**: push all the way down, hit the
base case, then pop all the way back up doing nothing extra on the way.
Every popped frame was "empty" — it had already finished its own work
(the `printf`) before it pushed the next one.

## Case 2: `mystery` and `factorial` — a stack that holds unfinished work

```c
int mystery(int n) {
    if (n == 1)
        return 1;
    return n + mystery(n - 1);
}
```

This is where it gets more interesting. Each frame pushes **holding an
unfinished expression** — it can't complete `n + ???` until the frame
above it pops with an answer.

Tracing `mystery(3)`:

```text
PUSH mystery(3)   holding: 3 + ???     stack: [3]
PUSH mystery(2)   holding: 2 + ???     stack: [3, 2]
PUSH mystery(1)   base case, returns 1 stack: [3, 2, 1]
POP  mystery(1)   returns 1            stack: [3, 2]
                  mystery(2) resolves: 2 + 1 = 3
POP  mystery(2)   returns 3            stack: [3]
                  mystery(3) resolves: 3 + 3 = 6
POP  mystery(3)   returns 6            stack: []
```

Compare this to `countdown`: here, **popping isn't the end of the
story** for the frame underneath. Each pop hands a value up to the
frame still waiting, and that frame uses it to finish its own unfinished
expression before it, too, can pop.

`factorial` works identically, just with `*` instead of `+`:

```text
PUSH factorial(4)  holding: 4 * ???
PUSH factorial(3)  holding: 3 * ???
PUSH factorial(2)  holding: 2 * ???
PUSH factorial(1)  holding: 1 * ???
PUSH factorial(0)  base case, returns 1

POP factorial(0) -> 1
POP factorial(1) -> 1 * 1 = 1
POP factorial(2) -> 2 * 1 = 2
POP factorial(3) -> 3 * 2 = 6
POP factorial(4) -> 4 * 6 = 24
```

---

## The "abccba" / palindrome shape

You noticed this pattern yourself during the trace: going down
`4, 3, 2, 1` and coming back up `1, 2, 3, 4` mirrors, like a palindrome.

That's not a coincidence — it's a direct consequence of LIFO ordering.
Whatever pushes last, pops first. So the *order things finish in* is
always the exact reverse of *the order they were called in*. Push
sequence `4, 3, 2, 1` forces pop sequence `1, 2, 3, 4` — no other order
is possible with a strict stack.

```text
push order:  4  3  2  1
pop order:   1  2  3  4
             \__________/
              mirror image
```

---

## Why the call stack can overflow (PDA connection)

A PDA's stack isn't infinite in real machines — and neither is a
program's call stack. Every `PUSH` (function call) consumes real memory
for that frame's local variables. If the base case is never reached
(or is unreachable, e.g. `factorial(-1)` never hits `n == 0`), the stack
only ever grows:

```text
PUSH, PUSH, PUSH, PUSH, ... (never pops, base case unreachable)
```

Eventually there's no more room — a **stack overflow**, and the program
crashes. In PDA terms: an automaton that never transitions to a state
that lets it start popping will keep pushing until its stack (however
large) is exhausted.

## Side-by-side: PDA vocabulary <-> recursion vocabulary

```text
PDA term            Recursion term
-----------------    -----------------
push                 function call
pop                  function returns
top of stack         the currently running (innermost) call
stack symbol         a paused call, holding its local variables
                     and any unfinished expression
LIFO order           calls must finish in exact reverse
                     order of how they were made
stack overflow       missing/unreachable base case ->
                     stack grows forever -> crash
```

## What I need to remember

```text
a call stack is a real stack: strict push on call, strict pop on return

countdown-style recursion: pop with nothing left to do
  (all work happens BEFORE the recursive call)

mystery/factorial-style recursion: pop while holding unfinished work
  (work happens AFTER the recursive call returns, using its value)

push order and pop order are always mirror images of each other
  (LIFO guarantees this — it's not a coincidence)

no base case = pushes forever = stack overflow = crash,
  exactly like a PDA that never reaches a popping state
```
