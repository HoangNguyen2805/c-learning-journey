# 4.7 - Register Variables

## Big idea

`register` is an old C keyword that suggests to the compiler:

```text
Try to keep this variable in a fast place if possible.
```

Example:

```c
register int i;
```

This usually means the variable is used a lot, like a loop counter.

---

## Important

`register` is only a suggestion.

The compiler can ignore it.

```text
register does not guarantee the variable is faster
```

Modern compilers usually optimize code better by themselves, so `register` is rarely needed today.

---

## Where register can be used

`register` is usually used with:

```text
local variables
function parameters
```

Example local variable:

```c
int main() {
    register int i;

    for (i = 0; i < 10; i++) {
        printf("%d\n", i);
    }

    return 0;
}
```

Example function parameter:

```c
int stringLength(register char s[]) {
    int i;

    i = 0;

    while (s[i] != '\0') {
        i++;
    }

    return i;
}
```

---

## Rule about address

You cannot take the address of a `register` variable.

This is not allowed:

```c
register int x;
printf("%p\n", &x);
```

Why?

Because `register` means the variable might not have a normal memory address.

---

## K&R reason for register

K&R uses `register` because older compilers needed more help with optimization.

A programmer could mark variables that are used very often.

Example:

```c
register int c;
register int i;
```

This tells the compiler:

```text
These variables are used a lot.
Try to make them fast.
```

## Understanding & (address-of operator)

Every variable in C lives somewhere in memory.
Memory is like a long street of numbered houses — each house holds
one piece of data, and each house has a unique address number.

When you declare `int x = 5;`, C finds an empty house in memory,
puts 5 inside it, and labels that house `x`.

The `&` operator means "give me the ADDRESS of this variable" —
i.e., which house number does x live at?

---

## & behaves differently in printf vs scanf

### scanf — & is REQUIRED
```c
scanf("%d", &x);
```
scanf needs the ADDRESS of x so it knows WHERE to store the number
it reads. Without &, scanf has no way to find your variable in memory
to write into it.

### printf — & is NOT used for normal printing
```c
printf("%d", x);    // correct - give printf the VALUE of x
printf("%d", &x);   // wrong   - this gives printf the ADDRESS, not the value
```
printf just needs the VALUE to display — it doesn't need to know
where x lives in memory, just what's inside it.

### printf with %p — special case, printing an address itself
```c
printf("%p\n", &x);
```
`%p` is a special format specifier that prints a memory ADDRESS.
Here, passing `&x` is intentional — "show me where x lives in memory."

This is legal for normal variables, but ILLEGAL for register variables,
since register variables might not have a standard memory address at all
(they might live in a CPU register instead, which has no address).

---

## Summary: when to use & and when not to

```text
scanf + &x    = always required (scanf needs to know WHERE to write)
printf + x    = correct for normal value printing (give the value)
printf + &x   = wrong for normal printing (gives address, not value)
printf + %p + &x = correct ONLY when you specifically want to print
                   the memory address itself (advanced, rare)
register + &x = NEVER allowed - register variables may have no address
```

## Connection to Chapter 5 (Pointers)

This concept of "addresses" is exactly what Chapter 5 (Pointers) is
built on. A pointer is simply a variable that STORES a memory address.
When you write `&x`, you're getting the address of x - and that address
is exactly what a pointer would hold.

This is why scanf uses & - it's actually passing a pointer (address)
to the function, so the function can reach back and modify the original
variable. This is the same problem we hit in the Fibonacci practice when
enterNumber(a, b) couldn't modify main()'s variables - the real fix
(which we deferred) is pointers, exactly this concept.
